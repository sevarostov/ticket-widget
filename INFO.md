## Технические решения 

1. Архитектурные решения
   MVC + слои:

Контроллеры — только маршрутизация и передача данных.

Сервисы (App\Services) — бизнес‑логика (создание тикета, смена статуса, расчёт статистики).

Репозитории (App\Repositories) — работа с БД (фильтрация, пагинация).

Ресурсы (App\Http\Resources) — форматирование API‑ответов.

SOLID/DRY/KISS:

Единый сервис для тикетов (TicketService) с методами: createTicket(), updateStatus(), getStatistics().

Репозиторий TicketRepository с методами фильтрации (filterByDate(), filterByStatus()).

Переиспользуемые FormRequest для валидации.

PSR‑12:

Соблюдение стандарта кодирования PHP (именование, отступы, документация).

2. Технологический стек
   Laravel 12 — фреймворк для MVC, роутинга, API, Blade‑шаблонов.

PHP 8.4 — версия с поддержкой enum, readonly‑классов, улучшений производительности.

MySQL — основная БД (миграции, связи сущностей).

Spatie\laravel‑permission — управление ролями (роль «manager»).

Spatie\laravel‑medialibrary — загрузка, хранение, выгрузка файлов, связь с Ticket.

Carbon — работа с датами (периоды «день/неделя/месяц»).

Docker Compose — локальное окружение (PHP‑FPM, MySQL, Nginx).

PhpUnit — тестирование (юнит, фича).

Darkaonline/l5-swagger  Swagger/OpenAPI 3.0 — документация API  [GET /api/documentation]  

3. Модели и связи
   
User (users):

Поля: name, email, password.

Роль: «manager» (через spatie/laravel‑permission).

Customer (customers):

Поля: name, phone (E.164), email.

Уникальность: (phone, email) для ограничения создания тикетов в день.

Ticket (tickets):

Связи: customer (ManyToOne), files (через medialibrary).

Поля: topic, text, status (enum), date_responded_at.

File — через Media модели (medialibrary), привязка к Ticket.

4. Роли пользователей. В проекте реализовано ролевое управление доступом (RBAC) с использованием пакета Spatie Laravel Permission. Определены три роли с разными уровнями доступа к функционалу.
   - Роль ***admin***
Права:

полный доступ ко всем маршрутам в авторизованной зоне;

наследование ролей manager и hr (имеет все их права);

возможность обновлять статус тикетов (ticket.updateStatus);

доступ к просмотру тикетов (ticket.index, ticket.show).

Примеры доступа:

Доступ к списку тикетов:

маршрут: GET /ticket;

статус: 200 OK;

результат: отображается список всех тикетов.

Просмотр конкретного тикета:

маршрут: GET /ticket/{id};

статус: 200 OK;

результат: отображаются детали тикета.

Обновление статуса тикета:

маршрут: POST /ticket/{id}/status;

параметры: ['status' => 'processed'];

статус: 302 Redirect;

результат: статус тикета обновлён, редирект на предыдущую страницу.

Технические детали:

email: admin@example.com;

пароль: admin_pwd;

имеет роли: admin, manager, hr;

может выполнять любые действия в авторизованной зоне.

- Роль ***manager***
  
Права:

доступ к просмотру списка тикетов (ticket.index);

доступ к просмотру конкретных тикетов (ticket.show);

нет прав на обновление статуса тикетов.

Примеры доступа:

Доступ к списку тикетов:

маршрут: GET /ticket;

статус: 200 OK;

результат: отображается список тикетов для просмотра.

Просмотр конкретного тикета:

маршрут: GET /ticket/{id};

статус: 200 OK;

результат: отображаются детали тикета для просмотра.

Попытка обновления статуса тикета (запрещено):

маршрут: POST /ticket/{id}/status;

параметры: ['status' => 'processed'];

статус: 302 Redirect;

сообщение: «Недостаточно прав для совершения действия»;

результат: статус не изменён, редирект на предыдущую страницу.

Технические детали:

email: manager@example.com;

пароль: manager_pwd;

имеет роль: manager;

разрешения: ticket.

-  Роль ***hr***

   Права:

доступ к главной странице (home);

доступ к разделу резюме (resume);

нет доступа к маршрутам тикетов.

Примеры доступа:

Доступ к главной странице:

маршрут: GET /home;

статус: 200 OK;

результат: отображается главная страница.

Попытка доступа к списку тикетов (запрещено):

маршрут: GET /ticket;

статус: 302 Redirect;

сообщение: «Недостаточно прав для совершения действия»;

результат: редирект на главную страницу или предыдущую.

Попытка просмотра тикета (запрещено):

маршрут: GET /ticket/{id};

статус: 302 Redirect;

сообщение: «Недостаточно прав для совершения действия»;

результат: редирект без отображения данных тикета.

Попытка обновления статуса (запрещено):

маршрут: POST /ticket/{id}/status;

статус: 302 Redirect;

сообщение: «Недостаточно прав для совершения действия».

Технические детали:

email: hr@example.com;

пароль: hr_pwd;

имеет роль: hr;

5. Валидация и безопасность
   
FormRequest:

StoreTicketRequest — валидация телефона (регулярное выражение для E.164), email, обязательных полей.

UpdateTicketStatusRequest — проверка статуса (new/in_progress/processed).

Ограничение создания тикетов в день:

Использована транзакция при сохранении тикета клиента в бд:  для ограничения создания не более одного тикета в день с одного номера/электронной почты.
 
В сервисе TicketService проверка: поиск тикета за последние 24 ч по phone или email.

Возврат ошибки 429 (Too Many Requests), если лимит превышен.

CSRF/CORS:

Защита форм (CSRF‑токен в виджете).

CORS‑политика для API (домены виджета).

6. API
   Маршруты:

POST /api/tickets → TicketController@store (создание тикета).

GET /api/tickets/statistics?period=day → TicketController@statistics (статистика).

Ресурсы:

TicketResource — ответ для тикета (включая customer, files).

TicketStatisticsResource — структура статистики (по периодам, статусам).

Элокуент‑скоупы:

forPeriod($period) в модели Ticket — фильтрация по периодам (Carbon).

7. Админ‑панель (Blade)
   Маршруты:

/admin/tickets — список тикетов (с фильтрацией).

/admin/tickets/{id} — детальная страница (тикет + файлы).

Фильтрация:

Форма с полями: дата (от/до), статус, email, phone.

Передача параметров в репозиторий (TicketRepository@filter()).

Смена статуса:

Submit form запрос на PATCH /api/tickets/{id}/status.

8. Виджет (Blade + AJAX)
   Маршрут: /widget → WidgetController@show.

Форма:

Blade‑шаблон с полями: имя, телефон, email, тема, текст, файлы.

Стилизация (CSS) для встраивания в `<iframe>`.

AJAX:

Отправка на /api/tickets.

Обработка ошибок (422, 429) и успех (201) с отображением сообщений.
